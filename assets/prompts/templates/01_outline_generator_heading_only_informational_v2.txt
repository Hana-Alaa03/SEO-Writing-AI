  2. Components/How Each Works: Explain the main parts, mechanisms, or operating model of each side when relevant.

  4. Pros & Cons: Cover advantages and limitations of each side separately when the topic requires it.
  5. Use Cases & Scenarios: Explain when to use X alone, when to use Y alone, and when both can work together.
  6. Hybrid Strategy: Cover how the two options can support each other when the comparison topic naturally involves combining them.
  7. Implementation or Application: Show how a reader can apply, start, combine, or evaluate the options.
  8. Measurement or Success Indicators: Include metrics, evaluation criteria, or signs of success when relevant.
  9. Mistakes: Include common mistakes in choosing, combining, or measuring the compared options.
  10. FAQ: Common misconceptions and choosing advice.

Comparison sections are OPTIONAL.

Comparison is a decision pattern, not always a section.

Use a standalone comparison H2 ONLY if:
- there are clear, specific alternatives the user must choose between
- the comparison itself is a major decision step

### 5. HEADING QUALITY RULES

- **Separate Intents Clearly**: Do not overlap sections. If 'Features' and 'Comparison' answer the same question, merge them or remove one. 
- **Split High-Intent Needs**: Do not group standalone search needs (e.g., location, opening hours, tickets) into one generic H2 like "Visitor Information". Give each major practical need its own H2 if commonly searched independently.
- Prefer single-purpose headings.
- Avoid combining practical informational intent with editorial enrichment such as "تجارب الزوار", "تجربة مميزة", or "تجربة متكاملة" unless it reflects a real search intent.

#### INTRO VS FIRST H2 DISTINCTION RULE
- The introduction provides hook, interest, and baseline framing only.
- The introduction must not be the definition.
- The first visible H2 must be the definition/context section for beginner and destination topics.
- If the intro label or first H2 both perform the same definition role, rewrite the intro as a hook and keep the definition as H2.

#### CONCLUSION BEHAVIOR RULE
- The final H2 should provide a clear takeaway, practical next step, recommendation, or application.
- Avoid generic endings such as “Conclusion” or “Final Thoughts”. The final H2 must signal the takeaway.
- The conclusion may contain one soft, helpful next step, but must remain non-promotional.

#### SERP GROUNDING RULE
- When `serp_outline_brief` is provided, prioritize its observed topics and `heading_generation_guidance` over generic structural assumptions.
- Use the `must_consider_sections` list as a mandatory baseline for essential reader info.
- Do NOT invent data (PAA, related, facts) if the brief indicates they were not observed.
- Ensure the outline reflects the `observed_page_type` (e.g., if SERP shows a "venue guide" pattern, stay closer to EXPERIENCE-BASED flow).

- If there is an H2 about tickets/pricing/booking, do not add an FAQ asking the same basic ticket or booking question.
- If there is an H2 about activities, do not add an FAQ asking what activities are available.
- If there is an H2 about location/access, do not add an FAQ asking how to reach the place.
FAQ must cover only edge cases, practical variations, or unanswered visitor concerns.

2. Do not create both a working-hours H2 and a best-time/planning H2 unless they answer clearly different needs.
- Working hours = official opening/closing times only.
- Best time/planning = crowd avoidance, ideal days, visit duration, route, or suitability.
- If the best-time section only repeats timing information, remove it or merge it into the working-hours section.
- Do not create both a planning H2 and a tips H2 if they answer the same visit-preparation need.
- Planning = timing strategy, duration, route, crowd avoidance.
- Tips = preparation, rules, what to bring, common mistakes.
If they overlap, merge or rewrite one.

3. If `brand_name` is provided, include exactly one contextual brand mention only when the topic supports it.
- For event/venue/ticket/access topics, the brand mention may be a booking, ticketing, pricing, or access utility.
- For strategy/service/implementation topics, the brand mention must be about applying or implementing the strategy, not booking.
- For pure knowledge topics with no practical action or implementation angle, do not include a brand mention.
- Prefer placing it as a practical H3 under the most relevant task H2 when that H2 naturally supports subheadings.
- If no suitable H3 placement exists, include it as one non-promotional FAQ question.
- Do not create a standalone brand H2.
- Do not use promotional wording.
- Do not use “why choose [brand]”, “best platform”, “book now”, or sales CTA language.
- Do not replace the exact brand name with generic wording such as "المنصة الرسمية", "منصة الحجز", or "الموقع الرسمي".
- If FAQ count is limited, replace the weakest generic question with the contextual brand FAQ instead of adding a duplicate.
Preferred Arabic event/venue H3 pattern: "طريقة حجز تذاكر [entity] عبر [brand_name]"
Preferred Arabic event/venue FAQ pattern: "هل يمكن حجز تذاكر [entity] عبر [brand_name]؟"
Preferred Arabic strategy FAQ pattern: "كيف تساعدك [brand_name] في تنفيذ استراتيجية [topic] متكاملة؟"
Forbidden outside event/venue/ticket topics: "هل يمكن حجز تذاكر [educational topic] عبر [brand_name]؟"

4. The conclusion H2 must be practical, not editorial.
Avoid: "تجربة زيارة", "استكشاف المزيد", "خطواتك القادمة", "خلاصة وتجربة".
