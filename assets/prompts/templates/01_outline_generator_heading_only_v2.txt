### 5. HEADING QUALITY RULES

#### FEATURES VS COMPARISON SEPARATION RULE

- If the offer section answers "how to choose" or overlaps with evaluation criteria, it is INVALID and must be rewritten as concrete services.
- Offer sections define what services/deliverables/options are available.
- Features sections MUST define evaluation criteria only (HOW to evaluate).
- They MUST describe how the user evaluates or chooses between options.
- They MUST NOT include direct comparisons (A vs B) or specific alternatives.
- Offer and features MUST NOT answer the same question. If both headings are about choosing/evaluating the best provider, the offer section is missing and must be rewritten as concrete services/deliverables.

- Comparison sections MUST compare specific options, types, or scenarios (WHAT to choose between).
- They MUST apply real decision criteria to help the user choose.
- They MUST NOT restate general evaluation criteria in abstract form.

### CONFLICT PREVENTION

- If comparison would only repeat the features or offer section, it MUST be embedded or omitted, not promoted to a standalone H2.
- If Features and Comparison sections overlap conceptually, the output is INVALID and must be regenerated with clearer separation.
#### SECTION ANGLE SEPARATION RULE

- If multiple sections perform segmentation, each MUST use a different decision dimension.
- Offer sections usually segment by type, category, deliverable, product/service option, or user segment.
- Location sections segment by area, geography, proximity, accessibility, or local suitability.
- Pricing/proof sections segment by cost driver, pricing model, package, scope, type, or location when relevant.
- Features sections define evaluation criteria, not option groups.
- Process sections explain action sequence, not choices.

If two H2s could be summarized with the same question, the output is weak.

Examples:
- Offer answers: "What options exist?"
- Features answers: "How should I evaluate?"
- Location answers: "Where is best for my need?"
- Pricing answers: "What will it cost and why?"
- Process answers: "How do I complete the action?"
- FAQ answers: "What common questions remain?"

If two sections use the same decision dimension or repeat the same conceptual role, the output is INVALID and must be regenerated with clearer separation.

- Avoid repeating the same instruction across multiple sections.
- Each section should follow the GLOBAL RULE instead of redefining it.
- Each H2 must represent a unique decision angle:
  - Criteria
  - Comparison
  - Pricing
  - Segmentation
  - Process
  - FAQ
  - Next step

### 7. HEADING CALIBRATION EXAMPLES
- **Generic vs. Decision-Focused (Features/Criteria):**
  - Bad: "أهم المزايا والمواصفات"
  - Bad: "المزايا التي تساعدك على اختيار [entity] مناسب"
  - Good: "معايير اختيار [entity] المناسب لـ [action] في [area]"
  - Good: "أهم المواصفات التي تحدد جودة [entity] قبل [action]"

- **Descriptive vs. Decision-Based (Location):**
  - Bad: "توزيع [entity] في [area]"
  - Bad: "أفضل مناطق [area] للبحث عن [entity]"
  - Good: "أفضل مناطق [area] لاختيار [entity] حسب احتياجك"
  - Good: "أفضل مناطق [area] لـ [action] [entity] بناءً على الميزانية ونمط الاستخدام"
  - Good: "كيف تختار المنطقة الأنسب لـ [action] [entity] في [area]؟"

- **Generic vs. Action-Anchored (Process):**
  - Bad: "رحلة البحث عن [entity] المثالي"
  - Good: "خطوات [action] لـ [entity] في [area] عبر [Brand]"

- **Vague vs. Specific (FAQ):**
  - Bad: "أسئلة شائعة حول [area]"
  - Good: "أسئلة شائعة حول [entity + action + area]"
